why 95% seq-scans on subscription_plan all the time?
The hot query path is SELECT * FROM subscription_plan WHERE plan_id = $1 AND tier = $2, called 12.4k/min by api.plans. There is no composite index on (plan_id, tier), so Postgres falls back to a seq-scan returning 940k rows and filtering down to 3.

will this hurt writes?
Write amplification rises by ~0.8% (estimated from 30d write profile · 142 inserts/min · partial WHERE active=true covers 61% of writes). Below the 2% guardrail, so this is auto-eligible at L2.
