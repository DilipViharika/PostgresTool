# FATHOM — Postgres Schema Diff

The schema-diff service snapshots a Postgres connection's schema into a
normalised JSON shape, diffs two snapshots deterministically, and generates
a best-effort migration script for review. It reads `information_schema` and
`pg_catalog` only — no extensions required, works on Postgres 12+.

Every piece is **preview only** — the service never executes DDL. An admin
reviews the generated SQL and runs it against their DB themselves.

## Endpoints

All routes gated by `authenticate` + `resolveWorkspace`.

### `POST /schema/snapshot` (viewer)

```json
{ "connectionId": "prod-primary", "schema": "public" }
```

Returns the snapshot JSON (tables / columns / indexes / PKs / FKs / views /
sequences / enums). Keys are sorted alphabetically so the same schema always
produces byte-identical output — safe to diff in CI.

### `POST /schema/diff` (viewer)

```json
{
    "leftConnectionId": "staging",
    "rightConnectionId": "prod-primary",
    "leftSchema": "public",
    "rightSchema": "public",
    "includeMigrationSql": true
}
```

Returns:

```json
{
  "diff": { "tablesAdded": [...], "tablesRemoved": [...], "tablesChanged": [...], ... },
  "summary": "3 tables added, 1 changed (2 column changes), 2 indexes added",
  "migrationSql": "BEGIN;\n-- ...\nCOMMIT;"   // only if includeMigrationSql is true
}
```

### `POST /schema/diff/migration` (editor)

Accepts a diff payload directly (perhaps hand-edited in the UI) and
regenerates the migration SQL.

```json
{ "diff": { … }, "direction": "left-to-right" }
```

## Diff shape

```
{
  tablesAdded:    [{ name, table }]
  tablesRemoved:  [{ name }]
  tablesChanged:  [{
    name,
    columnsAdded:   [{ name, dataType, ... }],
    columnsRemoved: [{ name }],
    columnsChanged: [{ name, before, after, changes: { dataType?, isNullable?, columnDefault? } }],
    indexesAdded / indexesRemoved / indexesChanged,
    foreignKeysAdded / foreignKeysRemoved,
    primaryKeyChanged: null | { before, after }
  }]
  viewsAdded / viewsRemoved / viewsChanged
  sequencesAdded / sequencesRemoved / sequencesChanged
  enumsAdded / enumsRemoved / enumsChanged
  totalChangeCount
}
```

## Generated migration SQL

The generator emits a single transaction:

```sql
-- Generated by FATHOM schema-diff at 2026-04-22T14:12:03Z
-- REVIEW CAREFULLY before running against a production database.

BEGIN;

CREATE TABLE "public"."audit_log" (
    "id"         BIGSERIAL NOT NULL,
    "created_at" TIMESTAMPTZ NOT NULL DEFAULT now(),
    "actor_id"   INTEGER,
    PRIMARY KEY ("id")
);

ALTER TABLE "public"."users" ADD COLUMN "timezone" TEXT;

ALTER TABLE "public"."orders"
    ALTER COLUMN "amount_cents" TYPE BIGINT
    USING "amount_cents"::BIGINT;  -- review USING clause

CREATE INDEX CONCURRENTLY "orders_user_id_idx" ON "public"."orders" ("user_id");

ALTER TYPE "public"."order_status" ADD VALUE 'refunded';
-- WARNING: removing enum values is not supported by PostgreSQL; manual work needed.

COMMIT;
```

Handled operations:

- `CREATE TABLE` with columns, NOT NULL, PK.
- `DROP TABLE`.
- `ALTER TABLE ADD / DROP COLUMN`.
- `ALTER COLUMN TYPE` with a `USING` stub and review warning.
- `CREATE INDEX CONCURRENTLY` / `DROP INDEX CONCURRENTLY`.
- `ALTER TABLE ADD / DROP CONSTRAINT` for foreign keys.
- `CREATE / DROP VIEW`.
- `CREATE / DROP SEQUENCE`.
- `ALTER TYPE … ADD VALUE` for enum additions.
- Enum **removals** emit a WARNING comment (Postgres has no native support).

All identifiers are double-quoted to tolerate mixed-case and reserved words.

## Safety

- No DDL is executed. The service only reads `information_schema` and
  `pg_catalog`, then returns generated SQL.
- `information_schema` is restricted to objects the connecting role is
  granted on — the service cannot reveal schemas the user's monitored
  connection doesn't already have access to.
- Type changes that require data conversion emit a `USING` stub and a
  review comment rather than a bare `ALTER` that might corrupt data.
- Enum value removals are not attempted; a warning is emitted instead.
